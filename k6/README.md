# 🧪 k6 Load Tests — Stellar MarketPay API

Production-like load tests for the three core read/write endpoints, with SLA
enforcement, CI automation, and week-over-week trend reporting.

```
k6/
├── README.md                  ← you are here
├── config.js                  ← shared SLA thresholds, ramp profile, helpers
├── lib/
│   ├── helpers.js             ← data-gen utilities (Stellar keys, query builder)
│   └── trend-report.js        ← compares current vs. previous run → Markdown + JSON
├── seed-data.js               ← Node script: seeds jobs + profiles → test-fixtures.json
├── get-jobs.js                ← GET  /api/jobs
├── get-profiles.js            ← GET  /api/profiles/:publicKey
├── post-applications.js       ← POST /api/applications
├── test-fixtures.json         ← generated by seed-data.js (git-ignored)
└── results/                   ← k6 summaries + trend reports (git-ignored)
```

---

## SLA targets

| Gate | Target | Load |
|------|--------|------|
| Latency | p(95) **< 200 ms** | 100 VUs |
| Reliability | **0 % error rate** (< 0.1 % enforced) | 50 VUs |

Every script uses the same ramp profile (`config.rampingStages`) so one run
validates **both** gates:

```
0 → 10 VUs  (warm-up)
   → 50 VUs  (held: validates 0 % error rate)
   → 100 VUs (held: validates p(95) < 200 ms)
   → 0       (ramp-down)
```

Thresholds are enforced by k6 — a run **exits non-zero** if the SLA is broken.

## Run locally (Docker)

The easiest path mirrors CI exactly — a throwaway stack + the official k6 image:

```bash
# 1. Start a production-like stack (Postgres + Redis + backend, tuned for load)
docker compose -f docker-compose.loadtest.yml up -d --build

# 2. Seed deterministic test data (writes k6/test-fixtures.json)
JWT_SECRET=loadtest-jwt-secret-with-sufficient-length-for-signing \
K6_BASE_URL=http://localhost:4000 \
  node k6/seed-data.js

# 3. Run a script (run from the k6/ directory so fixture/result paths resolve)
docker run --rm --network host -v "$PWD/k6":/work -w /work \
  -e K6_BASE_URL=http://localhost:4000 \
  grafana/k6:latest run get-jobs.js

# 4. Compare to a previous run (optional)
node k6/lib/trend-report.js --current k6/results --previous k6/previous-results

# 5. Tear down
docker compose -f docker-compose.loadtest.yml down -v
```

> **k6 must run with its working directory set to `k6/`** (the `-w /work` flag
> above mounts `k6/` to `/work`). Relative fixture and result paths resolve from
> there.

### Run with a local k6 binary

```bash
# install k6:  https://k6.io/docs/get-started/installation/
K6_BASE_URL=http://localhost:4000 JWT_SECRET=<same-as-backend> node k6/seed-data.js
cd k6 && K6_BASE_URL=http://localhost:4000 k6 run get-jobs.js
```

## The endpoints under test

| Script | Method & path | Notes |
|--------|---------------|-------|
| `get-jobs.js` | `GET /api/jobs` | Randomised category/search filters; exercises the cached read path. |
| `get-profiles.js` | `GET /api/profiles/:publicKey` | Fetches seeded profiles; hits the 5-min profile cache. |
| `post-applications.js` | `POST /api/applications` | Writes. Uses an injective `(VU, iteration) → freelancer` mapping so the `UNIQUE(job_id, freelancer_address)` constraint is never violated → 0 % errors. |

### Why seeding is required

`POST /api/applications` needs (a) **open jobs** and (b) **freelancer profiles**
that exist (FK constraint) and are **never reused** (unique constraint).
`seed-data.js` creates both and writes `test-fixtures.json`, which the scripts
load via `SharedArray`. Defaults: 30 jobs, 5000 freelancers (tune via
`SEED_JOBS` / `SEED_FREELANCERS`).

## Load-test tuning (why rate limits are relaxed)

Every API route sits behind per-IP rate limiters (e.g. `POST /api/applications`
allows 5 req/min/IP). At 50–100 VUs sharing one IP these would return `429`s and
mask the API's true throughput. Two **opt-in, default-preserving** env vars let
the load-test environment relax throttling and widen the DB pool:

| Env var | Default | Load-test value | Effect |
|---------|---------|-----------------|--------|
| `RATE_LIMIT_SCALE` | `1` | `1000` | Multiplies every rate-limit ceiling. Production keeps strict limits. |
| `DB_POOL_MAX` | `10` | `50` | Widens the Postgres pool so 100 concurrent VUs don't queue. |

Both default to production values, so existing deployments are unchanged.

## CI automation

`.github/workflows/load-test.yml`

- **Schedule:** every Monday 03:00 UTC, plus `workflow_dispatch` and on pushes
  to `main` that touch the load-test stack.
- Brings up the stack, seeds, runs all three scripts, and enforces the SLA.
- Uploads `k6/results/` + `k6/logs/` as the **`load-test-results`** artifact
  (90-day retention).
- Generates a **trend report** (`results/trend-report.md`) comparing p(95),
  error rate and throughput against the previous run, posted to the job summary.
